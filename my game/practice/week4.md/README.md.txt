Q1 원형 vs AABB: 모서리 부분에서 어떤 차이?
1.원형의 경우에는 원형이지만 돌아가진 않고, 비용을 덜 먹기 때문에 동그란 형태의 물건에 그 물건이 돌아가던,  돌아가지 않던간에 해당 물건이 원형이기만 한다면 넣어야 할 것 같았고, AABB의 경우 원형과는 달리 네모난 사각형이기 때문에 대각선에서 원형보다도 충돌이 더욱 쉬울 것 같았다.
떄문에 모서리의 부분을 약간 뭉특하거나 한다면 여과없이 원형을 넣어야 할 것 같다는 생각을 했으나, AABB가 계산을 매우 적게 먹기 떄문에 떄에 따라서 AABB와 원형을 잘 혼합하는 것이 중요하다고 생각했다.

Q2 AABB vs OBB: 회전시 무슨 차이?
2.AABB의 경우에는 회전시 회전이 되지 않기 떄문에 변하는 것이 없었다. 그러나 OBB의 경우 회전을 할 수 있기 때문에 정교한 충돌 처리를 위해서는 OBB를 이용 하는것이 옳다고 생각했다.

Q3 어떤 방식이 내 게임에 더 적합할까?
3.AABB가 내 게임에 더욱 적합 할 것 같았다. 후에 플랫포머 게임을 만들 것인데 플랫포머 게임들은 대다수가 회전을 한다거나, 투사체를 제외하고는 원형이 없기 때문에 AABB를 통해 최적화를 하는 것이 중요 한 과제라고 생각했으며, 이렇게 한다면 적을 꽤 넣는다 하더라도 무슨 문제가 없을 것 같다고 생각했다.

Q4 AI에게 물어본 것 중 흥미로웠던 답변.
4.왜 math.sqrt를 쓰는가?

원형 충돌 감지는 두 점 사이의 실제 거리를 구해서 판단하기 때문이야.

두 원의 중심 좌표가 있을 때:

dx = x1 - x2
dy = y1 - y2

이걸로 거리 계산:

distance = math.sqrt(dx*dx + dy*dy)

이 공식은 바로 피타고라스 정리야.

🎯 충돌 판정 원리

두 원이 충돌하려면:

중심 거리 < 반지름1 + 반지름2

즉:

distance < (r1 + r2)

그래서 **"실제 거리(distance)"**가 필요하고, 그걸 구하려고 sqrt를 쓰는 거야.

❗ 그런데 사실 sqrt 없어도 된다

실제로는 sqrt가 연산 비용이 조금 비싸기 때문에 보통 이렇게 최적화해:

dx = x1 - x2
dy = y1 - y2

distance_squared = dx*dx + dy*dy
radius_sum = r1 + r2

if distance_squared < radius_sum * radius_sum:
    collision = True

👉 핵심:

sqrt를 안 쓰고
양쪽을 제곱해서 비교
🔍 정리
math.sqrt는 실제 거리 계산을 위해 사용
하지만 충돌 판정만 할 거면:
❌ 굳이 sqrt 필요 없음
✅ 제곱 비교가 더 빠름
👍 언제 sqrt를 써야 하나?

sqrt가 필요한 경우:

실제 거리 값을 화면에 출력할 때
거리 기반 로직 (예: 가까울수록 데미지 증가)

필요 없는 경우:

단순 충돌 체크 (← 지금 상황)

위 sqrt에 대한 질문이 꽤나 흥미로웠었던 것 같았다. 왜냐하면 이제서야 내가 여태까지 배운 "게임을 만들려면 수학을 잘해야 한다." 라는 말이 피타고라스의 정리나 중심 거리, 같은 것들을 통해 증명이 됐기 때문이었다.
여태까지는 이론으로는 알고 있었으나, 직접적으로는 한 적이 없었기 때문에 꽤나 와닿지가 않고 있었던 것이 방금의 질문과, 비록 AI를 통해서긴 하나 이러나, 저러나 내가 직접 게임의 기초를 다지는 것임은 분명 한 사실이었기 때문에 꽤나 와닿은 것 같았다. 그 떄문에 위 질문이 가장 흥미로웠었던 것 같다.
